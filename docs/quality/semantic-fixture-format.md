# Language-Neutral Semantic Fixture and Corpus Format

- Status: Accepted schema and executable oracle baseline
- Fixture schema: `helix.semantic-fixture/1`
- Corpus manifest schema: `helix.semantic-corpus/1`
- Semantic profile: `helix-native-v1`
- Last updated: 2026-07-10
- Owner: Query semantics owner
- Plan items: `P01-018`–`P01-020`
- Governing requirements: `INV-002`, `INV-007`, `DATA-001`, `DATA-002`, `QUERY-001`, `QUERY-002`
- Governing gate: `G01`
- Decision: [ADR 0011](../adr/0011-use-tagged-json-semantic-fixtures.md)

This document defines the portable fixture envelope that every reference, optimized, host, protocol, SDK, GPU, persistence, and compatibility implementation consumes. It separates JSON structure validation from Helix semantic validation and represents every logical bit/type/order/error/state expectation without relying on a host language's number, map, exception, or iteration behavior.

Normative machine schemas:

- [semantic-fixture-v1.schema.json](../../fixtures/semantic/schema/semantic-fixture-v1.schema.json)
- [semantic-corpus-manifest-v1.schema.json](../../fixtures/semantic/schema/semantic-corpus-manifest-v1.schema.json)

Schema smoke/negative cases and their expected validation layer are indexed in [the examples README](../../fixtures/semantic/schema/examples/README.md).

## Goals and boundaries

The format must carry:

- deterministic initial documents, indexes/options, clocks, random bytes, transaction sequence, features, and quotas;
- normalized command trees, exact raw input bytes, or registered scalar/value operations;
- exact typed success values, row order and key tuples, mutation state, or structured `errors-v1` expectations;
- stable requirement/task/profile/tags for traceability and selection;
- source and RFC 8785 canonical SHA-256 hashes in a deterministic corpus manifest.

It is not HDoc, a public client protocol, a replacement for format golden vectors, executable user code, or permission to invent behavior in fixtures. Normative behavior comes from Specifications/accepted semantic documents; fixtures encode it. HDoc bytes/protocol captures may later be referenced or carried as `raw_input`, but retain their own format/version schemas.

## Directory and file convention

```text
fixtures/semantic/
  README.md
  schema/
    semantic-fixture-v1.schema.json
    semantic-corpus-manifest-v1.schema.json
    semantic-oracle-report-v1.schema.json
    check-semantic-examples.mjs
    examples/
  cases/                         # created/populated by P01-019
    <family>/<fixture-id>.json
  manifest.json                  # generated by P01-019
  oracle-report-v1.json          # generated independently by P01-020
reference/semantic-oracle/       # validator/interpreter/runner from P01-020
```

One fixture file contains one independently addressable scenario with one initial state and 1–256 sequential steps. File names are lowercase ASCII kebab/dot identifiers. The root `id` is globally unique and immutable; path moves update the manifest path but not semantic identity. Fixture IDs, step IDs, tags, requirements, plan items, collection names, and manifest entries must be unique in their defined scope and canonical-sorted where the format requires it.

Examples under `schema/examples/` validate the schema design and are not normative corpus members. Negative examples never appear in `manifest.json`.

## JSON parsing, canonicalization, and integrity

Fixture source is strict UTF-8 I-JSON:

- token-aware parsing rejects duplicate JSON object property names before a map can discard them;
- strings must contain valid Unicode scalar sequences; unpaired surrogates are rejected;
- trailing bytes, comments, NaN/infinity tokens, and non-JSON extensions are rejected;
- ordinary JSON objects are metadata/command structure; their source property order has no semantics;
- arrays always retain element order.

The schema dialect is pinned to [JSON Schema Draft 2020-12](https://json-schema.org/draft/2020-12). Schemas have URN IDs and only internal `$ref` values, so validation never fetches mutable/network schemas. A conforming validator first checks the schema itself against the pinned metaschema, then validates instances with the full Draft 2020-12 semantics.

Canonical fixture bytes are RFC 8785 JSON Canonicalization Scheme (JCS): recursively sorted object properties, canonical JSON primitives, no insignificant whitespace, then UTF-8. [RFC 8785](https://www.rfc-editor.org/rfc/rfc8785) constrains JSON numbers to interoperable binary64; therefore:

- Helix `int32`, `int64`, decimal coefficient/exponent, timestamp microseconds, date days, and logical sequence counters use canonical decimal strings where host precision could differ;
- `float64`, vector f16/f32 elements, binary, raw input, and random bytes use exact lowercase hexadecimal;
- bare JSON numbers are limited to safe bounded structural metadata/options (`[-2^53+1, 2^53-1]`) and never stand in for an ambiguous Helix logical number;
- non-finite logical floats remain exact bit strings inside ordinary valid JSON.

The corpus manifest records for each fixture:

```text
source_sha256    SHA-256 of exact committed source-file bytes
canonical_sha256 SHA-256 of RFC 8785 canonical UTF-8 bytes
bytes             exact source-file byte count
```

Source hashes catch editing/format drift; canonical hashes compare logically identical JSON across pretty-printers. The fixture does not contain its own hash, avoiding a circular value. The manifest itself is hashed by evidence/release artifacts rather than listing itself.

## Fixture envelope

Required root fields:

| Field | Meaning |
| --- | --- |
| `fixture_schema` | Exact `helix.semantic-fixture/1` discriminator |
| `id` | Stable lowercase ASCII scenario identity |
| `title`, optional `description` | Human purpose; never assertion logic |
| `requirements` | Canonical-sorted stable requirement IDs covered |
| `plan_items` | Canonical-sorted task/gate IDs that create/consume the case |
| `profiles` | Exact semantic/limit/collation/error/default-order versions |
| `tags` | Canonical-sorted selection/coverage labels |
| `initial_state` | Deterministic capabilities plus canonical collection state |
| `steps` | Ordered actions and exact expectations; state carries between steps |

Required v1 profiles are:

```json
{
  "semantics": "helix-native-v1",
  "limits": "limits-v1",
  "collation": "binary_utf8_v1",
  "errors": "errors-v1",
  "default_order": "default_order_v1"
}
```

Runners reject unknown required profile values before setup/action. They never substitute “current/latest” or infer a profile from package/fixture filename.

## Exact typed value representation

Every Helix value is a tagged JSON object. Stable tag `t` is case-sensitive.

| Logical state/type | Fixture payload |
| --- | --- |
| Missing (observable only) | `{"t":"missing"}` |
| Null | `{"t":"null"}` |
| Boolean | `{"t":"bool","value":true}` |
| int32/int64 | `{"t":"int32","value":"-1"}` / `int64`; canonical decimal string, checked range |
| float64 | `{"t":"float64","bits":"8000000000000000"}`; all exact 64 bits |
| decimal128 zero | `t`, `class:"finite"`, sign, coefficient `"0"`, exponent `"0"` |
| decimal128 nonzero | `t`, `class:"finite"`, sign, canonical 1–34 digit coefficient with no trailing zero, canonical exponent string |
| decimal128 special | `class:"infinity"` plus sign, or the one `class:"nan"` |
| String | `{"t":"string","value":"..."}`; scalar sequence preserved, no normalization |
| Binary | `{"t":"binary","subtype":0,"hex":"00ff"}` |
| Object | `{"t":"object","fields":[{"name":"x","value":...}]}` |
| Array | `{"t":"array","values":[...]}` |
| Timestamp | `{"t":"timestamp","microseconds":"..."}` |
| Date | `{"t":"date","days":"..."}` |
| UUID | canonical lowercase hyphenated text |
| ObjectId | canonical 24 lowercase hex digits |
| f16/f32 vector | `t:"vector"`, element, dimension, and one exact finite bit string per component |

Typed objects use a field-entry array because logical field presentation order is observable while JSON metadata object order is not. Field names must be unique exact strings and meet `limits-v1`; a duplicate is a semantic-lint failure. Typed arrays/objects cannot contain Missing. Missing is allowed only where an expression/path/expected observable state can produce it.

The JSON Schema validates tag shape, canonical spelling, component widths, and basic limits. Semantic lint additionally validates:

- int32/int64/timestamp/date numeric domains;
- decimal canonical cohort/exponent domain (`-6176 <= exponent` and `exponent + coefficient_digits - 1 <= 6144`);
- vector `dimension == bits.length` and finite component bits;
- valid scalar strings/names, UTF-8 byte/scalar limits, depth/count/size limits;
- unique object fields, root document `_id`, accepted ID types, semantic ID uniqueness/order;
- all cross-field rules that JSON Schema cannot express safely.

Schema-invalid logical values are malformed fixture files, not test cases for invalid database input. Invalid database values/JSON/UTF-8/duplicates are carried through a `raw_input` action as exact bytes so the fixture itself remains valid.

## Initial state and deterministic capabilities

`initial_state.collections` is a direct canonical state load, not a sequence of public inserts. Each document is a typed root object with one accepted unique `_id`; documents are strictly ascending under `default_order_v1`. This prevents fixture source order from becoming accidental insertion/natural order. Collection names are unique; collection arrays and index definitions follow their canonical schema order.

Optional `capabilities` supplies exact deterministic host inputs:

- `wall_time_reads` and `expiry_time_reads`: timestamp values consumed in call order;
- `monotonic_ticks` and `transaction_sequence`: canonical nonnegative integer strings consumed in call order;
- `random_hex`: exact byte chunks consumed in call order, never concatenated/rechunked by host convenience;
- `features`: explicit named Boolean capability availability;
- `quotas`: explicit safe-integer operational limits.

An unexpected capability read, exhaustion, different read size/order, or unused supplied value is a fixture failure unless the action contract explicitly declares optional consumption. No runner reads ambient time, randomness, locale, timezone, host feature, device, or quota while executing a deterministic semantic case.

Each step begins from the prior step's resulting state. A runner must reset the complete fixture sandbox between fixture files; state, clocks, randomness, IDs, caches, cursor records, and failures cannot leak across cases.

## Actions

### Structured command

```json
{
  "kind": "command",
  "command": {
    "find": "items",
    "filter": { "x": { "$eq": { "$value": { "t": "int64", "value": "1" } } } },
    "limit": 10
  }
}
```

The command tree uses ordinary JSON strings/Booleans/null/safe structural integers, arrays, and unique-property objects. Every logical literal is wrapped as exactly `{"$value": <logical_value>}`. This separates collection/path/operator/option syntax from data and prevents JSON inference from erasing int width, float bits, decimal, binary, time, identifier, object field order, or vector family.

The smoke command uses bare projection direction `1` because that is command grammar, not a stored logical int32. An unwrapped data literal is a semantic-lint failure even if the general recursive JSON Schema cannot know a future command position's grammar. `P01-020` owns the registered normalized-command/action grammar used by the oracle.

### Raw input

```json
{
  "kind": "raw_input",
  "target": "command",
  "encoding": "json",
  "compression": "identity",
  "bytes_hex": "7b"
}
```

Raw input covers malformed/truncated bytes, duplicate JSON properties, invalid UTF-8, unknown tags/versions, decompression limits, and future HDoc/protocol cases. Bytes are decoded exactly once under the named target/encoding/compression and the applicable raw/expanded limits. Runners never repair/re-encode the bytes before the component under test.

### Registered value operation

```json
{
  "kind": "value_operation",
  "operation": "value.compare",
  "arguments": [<observable_value>, <observable_value>],
  "options": {}
}
```

This action exercises scalar/path/hash/metric/cast/arithmetic semantics without smuggling them through an unrelated query. Operation IDs and argument/result/error contracts are versioned and enumerated by the independent oracle under `P01-020`; an unknown ID is a fixture-lint failure, not a database unsupported-operator case.

## Success expectations

A success requires:

```text
kind = success
value = exact observable typed value
order = explicit order expectation
state = explicit state expectation
```

The normalized success `value` is compared recursively by exact fixture representation, including type, bits, Missing, field presentation, array/row order, result counts, and metadata selected by the action contract. Implementations adapt their native response into this model; the fixture does not compare language class names, map order, exception text, or lossy JSON rendering.

## Error expectations

An error records normative `errors-v1` fields only:

```text
category
code
phase
outcome
retryable/scope/delay/attempt/token-presence
details exact/subset/absent
bounded causes
order
state
```

Human `message`, request/trace/error IDs, stack, localized text, and actual opaque retry-token bytes are deliberately absent. Retry token comparison is `absent`, `present`, or `ignore`. `details_match: exact` rejects extra/missing fields; `subset` requires the expected safe fields while allowing registered additive diagnostics; `absent` requires no client details. Cause order is exact and bounded.

Semantic lint verifies category/code prefix registration, retry safety, and outcome/state consistency:

- `not_committed` requires unchanged state;
- `committed` requires exact resulting state;
- `unknown` requires unknown state and never a guessed exact state;
- nonretryable requires scope `never`;
- unknown/future codes are not authored into a v1 expected case without a new registry/profile.

## Ordering expectations

Every step declares order even when no row stream exists:

```text
mode      exact | set | not_applicable
basis     default_order_v1 | explicit_sort | vector_rank |
          pipeline_ordinal | input_order | singleton |
          set_semantics | not_applicable
row_count exact nonnegative count
keys      one key tuple per exact row
```

For `mode: exact`, `row_count == keys.length` and the normalized result stream has the same count/order. An order key is an ordered component tuple:

- `value`: exact observable value plus `asc`/`desc` direction;
- `index`: nonnegative input/array/provenance index;
- `branch`: stable future combination/union source tag;
- `singleton`: stable one-row stage tag.

Structured tuples flatten the comparison/provenance sequence without exposing an engine's private cursor bytes. Default rows normally carry one ascending `_id` value component; explicit sort lists declared values plus hidden tie; unwind appends index; repeated sort includes the new key then prior components; group uses canonical semantic key; input-correlated results use index.

`mode: set` is allowed only for a response field whose versioned schema explicitly declares set semantics; current v1 document streams do not. It uses `basis: set_semantics`, an empty key list, and exact row count. `not_applicable` uses its matching basis, zero row count, and no keys. Tests never compare an ordered stream as a set for convenience.

## State expectations

Every step explicitly selects one:

| Mode | Meaning |
| --- | --- |
| `unchanged` | Complete state equals pre-step state, including no retry/cursor/derived publication promised by action scope |
| `exact` | Listed/all collection state equals typed canonical documents/options/indexes; scope says whether unlisted collections are ignored |
| `unknown` | Mutation outcome is genuinely unresolved; runner must not invent state and later recovery fixtures resolve it |
| `not_observed` | Action contract deliberately has no state oracle; prohibited for mutation/error cases that claim atomicity/durability |

Exact collections reuse the canonical initial-state representation and default ordering. State comparison is logical/canonical, not raw storage layout, unless a separate persistent-format fixture explicitly tests bytes.

## Semantic validation beyond JSON Schema

Validation runs in layers and stops at the first failing layer:

1. Strict UTF-8/I-JSON token parse with duplicate-key detection.
2. Pinned Draft 2020-12 schema validation.
3. Fixture semantic lint: domains, unique member IDs/names, canonical metadata order, profiles, operation registry, command literal positions, collection ID/order, vector/cardinality/state/error cross-fields, and `limits-v1`.
4. Runner capability/profile support check.
5. Scenario setup and action execution.
6. Exact value/order/error/state/capability-consumption comparison.

The dependency-free [example semantic checker](../../fixtures/semantic/schema/check-semantic-examples.mjs) demonstrates cross-field failures. It is evidence for the schema design, not the complete reference oracle. The full validator/oracle under `P01-020` must cover every rule here, emit stable fixture-diagnostic codes, and be used by all corpus consumers.

A malformed fixture is a harness failure and never counted as a passing expected database error. Runner reports include fixture/step ID, schema/profile, action layer, stable diagnostic, expected/actual hashes, skips, backend/host, and source commit—never only a Boolean summary.

## Independent reference oracle

The executable [`helix-reference-oracle/1`](../../reference/semantic-oracle/README.md) implementation is independent of the corpus generator, corpus integrity checker, optimized/runtime operators, planners, codecs, indexes, GPU kernels, and adapters. Its action executor never receives `step.expect`; expectations enter only the final comparison layer. Separate error, limit, operation, type, numeric, path, raw-input, command, and ordering logic is reconciled with the committed registries and semantic documents.

The oracle implements all 17 registered value operations, all current raw-input cases, normalized invalid-command precedence, and the populated reference `find` subset. It validates and executes every one of the 319 corpus steps in a fresh deterministic sandbox. Unit/property tests additionally prove exact mixed-numeric comparison, decimal rounding, total-order laws over representative ranks, strict JSON/UTF-8 behavior, time-offset bounds, path fan-out, vector scores, and stable diagnostics. Four mutation canaries independently alter expected value, error, order, and state and must be detected.

The deterministic [oracle report](../../fixtures/semantic/oracle-report-v1.json), validated by the [Draft 2020-12 report schema](../../fixtures/semantic/schema/semantic-oracle-report-v1.schema.json), binds the oracle/profile versions, corpus-manifest source hash, exact action/operation/outcome totals, ordered per-fixture observation hashes, zero skips, and verdict. It contains no wall time, duration, hostname, locale, random value, or mutable environment field.

This reference is an executable semantic baseline, not a production database. Compact boundary/error operations do not replace real resource allocation or fault injection, and successful mutation/aggregation/cursor/persistence/backend behavior remains assigned to later phases. Every later conformance run names the exact corpus manifest and oracle report hashes rather than regenerating expectations from the implementation under test.

## Corpus manifest

`helix.semantic-corpus/1` is generated deterministically from accepted files under `fixtures/semantic/cases/` and contains:

- exact fixture/schema/semantic/hash profile IDs;
- generator tool/version without wall-clock timestamp;
- exact fixture/step/success/error counts;
- requirement-to-fixture coverage arrays;
- entries sorted by fixture ID with path, source bytes/hash, canonical hash, requirements, tags, and step count.

Manifest semantic lint verifies unique/sorted IDs/paths/tags/requirements, entry hashes/bytes against files, root ID/profile agreement, exact aggregate counts, and coverage equality (not merely inclusion). Files absent from the manifest or manifest entries absent from disk fail. A selected subset report names the source manifest hash plus included IDs; it does not generate a misleading new “complete” corpus.

## Versioning and change policy

`helix.semantic-fixture/1` and `helix.semantic-corpus/1` are immutable contracts. Because schemas reject unknown properties, adding/removing/reinterpreting a field, tag, action, typed form, expectation, canonicalization, or cross-field rule requires a new schema identifier and explicit reader support. Adding new fixture instances under the same frozen semantics is additive.

Fixture/profile changes follow these rules:

- A legitimate new edge case gets a new globally unique ID and manifest entry.
- An accepted semantic behavior change gets a new semantic/schema/profile as applicable, new expected fixture IDs, ADR/matrix/migration impact, and preserves historical fixtures.
- A mistaken fixture expectation is not silently edited after gate acceptance; add a superseding case/change record and retain old evidence/hash history.
- Formatting-only changes alter source hash but not canonical hash; both remain visible and require ordinary review.
- Schema examples may evolve only by adding a new versioned example/schema baseline; negative expectations remain reproducible.
- Older runners reject unknown required schema/profile/action/type behavior, never reinterpret it.

## Security and resource rules

- Fixtures contain only synthetic/public data; never secrets, credentials, private/customer data, filesystem/host paths, or exploit payloads inappropriate for the repository.
- Raw/compressed inputs are bounded before decode/decompression/allocation and never executed as code.
- Command/value operation registries contain no JavaScript/Wasm/plugin source or arbitrary host callbacks.
- Schema validation disables network reference fetching and uses the committed schema hash.
- Paths are repository-relative, normalized, cannot escape the corpus root, and are never constructed from untrusted IDs without validation.
- Fixture/detail/report output follows `errors-v1` redaction; negative canaries use unmistakably synthetic markers.
- Huge limit cases use deterministic generators/compact raw artifacts where approved, with exact hash/size, rather than checking in impractical accidental expansions.
- Corpus selection, parallelism, timeouts, memory, and artifact output are bounded; parallel runners preserve manifest/report order.

## Required schema conformance

Before `P01-018` closes:

- both schemas parse and pass Draft 2020-12 metaschema checking;
- every `examples/valid` file passes schema and semantic example lint;
- every `invalid-schema` file fails the structural layer;
- every `invalid-semantic` file passes schema then fails its exact named semantic rule;
- all logical value branches, three action kinds, success/error, exact/not-applicable order, and state forms have schema coverage;
- canonical/source hash reproduction and local links/formatting pass;
- schemas/examples/checker are immutable-hashed in task evidence.

`P01-019` populates the normative [cases, manifest, and coverage ledger](../../fixtures/semantic/COVERAGE.md); `P01-020` implements the full independent validator/oracle. No optimized/runtime implementation may become the source of expected values.
